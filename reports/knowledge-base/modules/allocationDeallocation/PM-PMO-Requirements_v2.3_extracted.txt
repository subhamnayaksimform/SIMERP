1. Overview 
This document defines the requirements for introducing two new roles into SimERP: Project 
Manager (PM) and Project Management Office (PMO). These roles introduce a project-portfolio-
based access model alongside a structured allocation ticketing system. The system is ticket-driven 
by design: with the single exception of PMO direct allocation (§6.3), every allocation change flows 
through a ticket to reduce manual intervention and preserve traceability. 
PM assignment to projects is managed externally via the Zoho Analytics API. SimERP syncs project-
to-PM mappings from Zoho — there is no manual PM assignment UI within SimERP.

3. New Roles 
3.1 Project Manager (PM) 
A PM is responsible for managing one or more projects. Their visibility in SimERP is scoped to the 
projects they manage — they do not access department or team data. 
Key characteristics: 
• Each project has exactly one PM, sourced from the Zoho Analytics API sync 
• A PM can manage multiple projects simultaneously 
• PM can coexist with an org-hierarchy role (RM, TM, or EM) on the same employee — see §3.3 
• PM only has access to two pages: PM Projects and Requests 
• PMs cannot directly modify allocation records and cannot edit allocations inline — every 
change must be raised as a request (ticket) 
• Tickets belong to the project, not the PM. If the PM on a project changes (via Zoho sync), the 
new PM inherits all existing tickets for that project; the previous PM loses visibility 
3.2 Project Management Office (PMO) 
The PMO coordinates between PMs (who request developers) and the approvers — TMs and EMs 
(who supply them). They have organisation-wide visibility and own the allocation workflow.

Key characteristics: 
• Sees all projects and all employees across the organisation 
• Owns the allocation ticketing workflow — triages tickets, assigns departments, shortlists 
developers, drives approvals, and acknowledges de-allocations 
• Exclusive direct allocation: only PMO holds the “Add Allocation” permission to create a 
ProjectAssignment directly without a ticket (§6.3). PMs, TMs, and RMs are explicitly denied 
this action to maintain system integrity. Each direct allocation triggers an in-app + email 
notification to the project’s PM, the department’s TM/EM, the allocated employee’s RM, and 
the allocated employee 
• PMO is a standalone role — cannot be combined with RM, TM, or EM 
• Has full access within the PM_PROJECTS and ALLOCATION_TICKETS modules. 
• Acts as L1 support for PMs navigating the ticket types (operational note)

4.2.3 Replacement 
Replacement is presented to the user as a third option in the Create Request form. Internally it is 
stored as an Update Allocation (UA) ticket with a replacement flag, and the related replacement 
information is saved on that ticket — the system recognises only two ticket types (NA and UA). 
For a Replacement request, the allocation data is pre-filled from the existing assignment being 
replaced, and the following replacement fields are additionally available:

Replacement scope safeguard: replacement requests pre-populate the original developer’s 
department and tech stack as non-editable fields to ensure a like-for-like replacement. A note on the 
form states that changing the technology stack (a “scope swap”, e.g. replacing a QA with a Node.js 
developer) is not permitted through the replacement flow — it requires a fresh de-allocation plus a 
new standard NA ticket.

Ticket numbers are zero-padded 4-digit sequential integers per type (e.g., NA-0001, UA-0042). The 
sequence does not reset yearly. Stored internally as a ticketNumber Int column; the display string 
(e.g. NA-0001) is computed in the service layer by prepending the type prefix. Two PostgreSQL 
sequences (na_ticket_seq, ua_ticket_seq) provide atomic, gap-free numbering per type. Ticket detail 
URLs use the human-readable ID as the route parameter (e.g. /allocation-tickets/NA-0042) so URLs 
are shareable and meaningful to recipients. 
Tickets are never hard-deleted. Cancellation, archival, and any removal are implemented as soft 
delete (deletedAt) so the audit trail is preserved.

AWAITING_CONFIRMATION state.

5.3 On APPROVED — Automatic Allocation Change 
Whether a ticket reaches APPROVED via the two-party approval, PMO bypass, or PMO 
acknowledgement, the system automatically applies the change in a transaction: 
• NA approved → creates a new ProjectAssignment with status = GREEN and typeOfWork = 
DEVELOPMENT_WORK for any type of allocation. In case of ‘PR Reviewer’ type tickets, do 
not treat it as a project allocation — treat it as an ‘other allocation’ and add the developer as 
PR reviewer in that project. Other allocations count over the 100% allocation limit (they may 
exceed it). The originating requestType is persisted on the assignment. 
• UA (UPDATE_ALLOCATION) approved/acknowledged with % > 0 → updates the existing 
ProjectAssignment (only the fields whose values were changed; unchanged pre-filled fields are 
left as is)

• UA (UPDATE_ALLOCATION) approved/acknowledged with % = 0 (including dynamic Update 
Allocation de-allocations) → sets allocationEndDate = approval/acknowledgement date, then 
soft-deletes (archives) the ProjectAssignment 
Before applying, the system verifies the target ProjectAssignment still exists and deletedAt IS NULL. 
If the assignment was already deleted by another approved ticket, the transaction fails. 
If the transaction fails, the ticket reverts to its prior state (AWAITING_CONFIRMATION) and PMO is 
notified with a retry prompt. 
Validation 
To move a ticket from the AWAITING_CONFIRMATION state to APPROVED, approvals are 
required from two parties: 
• One PMO representative 
• The TM/EM of the selected department 
If PMO selects the Bypass checkbox, the TM/EM’s approval is not required — the PMO approval 
alone is sufficient. 
The ticket should clearly display which approvers have already provided approval and which 
approvals are still pending. If approval is bypassed by PMO then it should show the bypassed 
approver with separate indicator. 
If any modifications are made regarding the assignment to the ticket after approval has been 
provided by either party, all existing approvals should be invalidated, and the ticket should 
automatically revert to the AWAITING_CONFIRMATION state. 
5.4 Shortlist (NA tickets only, IN_REVIEW phase) 
During IN_REVIEW of a NA ticket, PMO and the TM/EM collaboratively add candidate developers to 
a working shortlist. PMO then selects one final developer, moving the ticket to 
AWAITING_CONFIRMATION. 
UA tickets have no shortlist — the developer is already identified via projectAssignmentId at creation 
time. 
The shortlist is not visible to the PM. The PM only sees the ticket state. 
The shortlist is ephemeral — once a developer is selected and the ticket moves forward, the shortlist 
data is no longer meaningful. 
Adding a developer to a shortlist shows a warning (but does not block) if that developer is already 
shortlisted on another active ticket. 
5.5 Bypass Approval (full flow only) 
PMO can bypass the TM/EM approval in emergency scenarios — when the Bypass checkbox is 
selected, the TM/EM’s approval is not required and the PMO approval alone moves the ticket to 
APPROVED (directly from IN_REVIEW). The TM/EM is notified when their approval is bypassed. No 
reason is required from PMO, but the bypass is recorded on the ticket for audit purposes via 
bypassedAt (timestamp) and bypassedById (FK to the PMO employee who triggered the bypass). 
Lighter-flow tickets have no approval to bypass. 
5.6 Approver Rejection (full flow only) 
The TM or EM of the assigned department can reject a ticket in AWAITING_CONFIRMATION. The 
ticket returns to IN_REVIEW for PMO to shortlist a different developer. No rejection reason is 
required. Lighter-flow (decrease) tickets cannot be rejected — de-allocations cannot be refused.

6.3 Direct Allocation (Add Allocation) 
PMO holds an exclusive “Add Allocation” action to create a ProjectAssignment directly on any 
project without raising a ticket. This is the only non-ticket allocation path in the system. 
• Available to PMO only — PMs, TMs, and RMs are explicitly denied (enforced via a dedicated 
CASL permission on the PMO role) 
• The creating PMO employee is recorded (createdById) for traceability 
• On creation, an in-app + email notification is sent to the project’s PM, the department’s TM/EM, 
the allocated employee’s RM, and the allocated employee

11. System Design Decisions 
These decisions were made during the design sessions (including Sprint 4 planning, 2026-06-03). 
Flag any disagreements before development begins. 
Decision Detail 
Partial update semantics UA forms are pre-filled with the current assignment values; the user 
updates only the fields requiring change. On apply, only fields whose 
values differ from the current assignment are updated — unchanged 
pre-filled fields are left as is. For Replacement requests, allocation 
data is pre-filled and replacement-specific fields are additionally 
shown (§4.2.3). 
Partial deallocation Updates ProjectAssignment.allocationPercentage in place (Option 
A). Full history is preserved via the ticket record itself. 
Auto-apply failure Ticket reverts to its prior state (AWAITING_CONFIRMATION). PMO 
is notified with a retry prompt. Failure is logged. 
Concurrent shortlist writes PostgreSQL atomic array operations handle concurrent PMO + 
TM/EM writes. Last-write-wins is acceptable for this low-frequency 
collaborative flow. 
Duplicate UA tickets If a PM attempts to create a UA for an employee+project combination 
that already has an active ticket (state ∈ OPEN, IN_REVIEW, 
AWAITING_CONFIRMATION), the request is blocked with a conflict 
error. For NA tickets (no specific developer at creation), no block 
applies. A shortlist-level warning is shown to PMO/TM/EM when 
adding a developer already shortlisted on another active ticket — this 
does not block the addition. 
CASL module names Two new permission modules: PM_PROJECTS (PM Projects page) 
and ALLOCATION_TICKETS (ticketing system). A dedicated PMO-
only permission governs direct allocation (§6.3). 
PM assignment No manual assignment UI in SimERP. All project-to-PM mappings 
are synced from Zoho Analytics API nightly at 12 AM using a new 
PROJECT_METADATA sync job. Zoho provides 
projectManagerEmail; the sync matches it to Employee.email to set 
Project.projectManagerId (FK). Unmatched emails leave 
projectManagerId as null and log a warning. PM changes in Zoho 
take effect in SimERP the following day (up to 24-hour lag is 
acceptable). 
Project.projectManager 
field 
The existing projectManager String? free-text field is replaced by 
projectManagerId BigInt? (FK → Employee) in the same schema

Ticket ID storage ticketNumber Int is stored on the AllocationTicket model. The display

Approval model Full-flow tickets in AWAITING_CONFIRMATION require two

Term Definition 
PM (Project Manager) Employee responsible for managing one or more projects. Visibility 
scoped to their projects only. Assigned via Zoho sync. 
PMO (Project Management 
Office) 
Organisation-wide role responsible for allocation coordination. 
Sees all projects and employees. Sole holder of direct-allocation 
access. 
Approver The TM or EM of the assigned department who approves or rejects 
a full-flow ticket in AWAITING_CONFIRMATION. An EM may 
approve for any department they manage. 
Allocation A ProjectAssignment record linking an employee to a project for a 
defined period and percentage. 
Fixed Allocation Standard percentage-based allocation (e.g. 100%, 50%, 25%). 
Dynamic Allocation Need-based allocation (e.g. maintenance projects). Implies 
maximum availability of 1–2 hours/day or 10–20 hours/month. 
Percentage field is disabled; ended via the Update Allocation 
action, which sets the allocation to 0% internally. 
Other Allocation A non-project allocation that marks a developer as a PR reviewer 
on a project. Other allocations count over (may exceed) the 100% 
allocation limit. 
New Allocation (NA) Request to add a new developer to a project. Ticket ID prefix NA-, 
enum NEW_ALLOCATION. 
Update Allocation (UA) Request to modify or end an existing developer’s allocation. Ticket 
ID prefix UA-, enum UPDATE_ALLOCATION. 0% = full 
deallocation. Replacement requests are stored as UA tickets with a 
replacement flag. 
Awaiting Confirmation Single confirmation state shared by both flows. Full flow: awaiting 
two-party approval (one PMO representative + the TM/EM; TM/EM 
waived on PMO bypass). Lighter flow: awaiting PMO 
acknowledgement; cannot be rejected. 
Replacement Request Presented as a third option in the Create Request form; stored as 
a UA ticket with a replacement flag, linked via Replacement Of. 
Permanent or Temporary. Department and tech stack are locked 
like-for-like. 
Allocation Accuracy Future analytics measuring the quality of initial allocations using 
replacement chains (e.g. repeated performance-driven 
replacements indicate low accuracy). 
Pending Actions (PMO) PMO dashboard section aggregating acknowledgements due, de-
allocation reminders, and failed auto-applies. 
Shortlist The working list of candidate developers during IN_REVIEW of a 
NA ticket. Ephemeral and not visible to PM. Does not apply to UA 
tickets. 
Developer Review PM’s permanent qualitative record about a developer on a project.
